What v2.1 adds

Version 2 made routing explicit and evidence-driven. Version 2.1 preserves the proven Opus planner and adds Fable as a deliberate, separately verified planning option.

1

Explicit model policy

Every agent declares a model alias, effort level, permission mode, and limited tool set.

2

Deterministic invocation

Use an explicit @agent-name mention when the test must call one named agent.

3

Honest runtime proof

Static validation proves files. A captured trace shows which agent and model were observed for one run.

4

Windows-native workflow

PowerShell instructions, safe backup steps, and both Bash and PowerShell tool availability where needed.

5

Stricter validation

Recursive scope checks, strict YAML, exact paths, policy enforcement, and secret-pattern scanning.

6

Deliberate permissions

Planning agents cannot edit. QA uses default mode so approved tests can run, while its prompt forbids intentional edits.

No magic and no guaranteed savings

Routing can improve the match between task and model. Actual quality, latency, and cost depend on prompts, project size, tool use, retries, provider behavior, and your plan. Measure your own workload.

Coordinator duties

  • Choose the lane from task risk, not prestige.
  • Send only the context the agent needs.
  • Name allowed files and forbidden actions.
  • Define evidence and a stop condition.
  • Own integration and user-facing claims.

Hard boundaries

  • Architect, Deep Reasoner, and Fable Planner are read-only planners.
  • Fable requires a version and availability preflight; use Opus when it is unavailable.
  • Fast Worker writes only inside the stated scope.
  • QA does not intentionally edit, but a test command can mutate artifacts.
  • QA shell access still requires user or session permission; host policy also matters.
  • Agent output is evidence to inspect, not authority to trust blindly.

Prefer the smallest sufficient lane

  1. Is the task fully specified and low risk? Try Fast Worker.
  2. Does it need structure or tradeoff analysis? Use Architect.
  3. Is the uncertainty itself the problem? Use Deep Reasoner.
  4. Want a second planning approach and Fable passed preflight? Explicitly opt in.
  5. Is the work complete and ready to challenge? Use QA Reviewer.

Escalate on evidence

  1. Stop a worker when scope is ambiguous.
  2. Escalate when constraints conflict.
  3. Do not turn retries into an unbounded loop.
  4. Return to the coordinator for a new decision.

Configured target: the model alias and effort written in an agent file. Observed route: the agent and model reported by one captured runtime trace. Neither is a cryptographic guarantee about future runs.

Agent playbooks

What each lane should return

01 / Architect
OPUS
High effortPlan mode

Return a decision-ready plan with boundaries, alternatives, risks, dependency order, and acceptance criteria. Do not edit files or run shell commands.

Good packet

"Design the auth boundary. Read only these three files. Compare two designs. Return a sequence and five acceptance checks."

02 / Deep Reasoner
OPUS
XHigh effortPlan mode

Resolve the hardest uncertainty. State assumptions, test competing explanations, identify disconfirming evidence, and recommend the next decision.

Good packet

"Explain the intermittent race. Read the trace and scheduler only. Rank hypotheses and propose the smallest discriminating test."

03 / Fable Planner (opt-in)
FABLE
XHigh effortPlan mode

After the user explicitly requests Fable and the environment passes preflight, turn a broad objective into a phased roadmap with scenarios and decision gates. Otherwise use Opus.

Good packet

"Plan a two-year platform transition. Read only the named evidence. Return scenarios, phases, decision gates, risks, and the first bounded handoff. Do not edit."

04 / Fast Worker
HAIKU
Low effortDefault mode

Make the named change in allowed paths, run exact checks, summarize the diff, and stop on missing scope or a consequential decision.

Good packet

"Rename this flag in two files. Do not touch generated code. Run the named unit test. Report changed paths."

05 / QA Reviewer
SONNET
High effortDefault mode

Inspect independently. Run only the exact command if user or session permission allows. Do not intentionally edit; note that test commands can mutate artifacts.

Good packet

"Review this diff against three criteria. Run exactly: python -m unittest -q. Do not edit. Report denied or unavailable tools."

Why effort differs: effort is a policy signal, not a quality promise. Low effort suits bounded execution. High and xhigh are reserved for planning, ambiguity, and adversarial review where additional reasoning can justify the latency.

Agent playbooks05

Testing

Deterministic invocation vs automatic routing

Test the named agent

Use an explicit @mention

When the question is "does this exact agent work?", explicitly invoke it in Claude Code. Natural-language hints may be ignored or routed differently.

> Use @agent-architect to inspect the project structure. Return three risks. Do not edit.
PS> claude -p "Return exactly FABLE_PREFLIGHT_OK. Do not use tools." ` --model fable --output-format stream-json --verbose --no-session-persistence

Verify that direct trace with verify_direct_model_trace.py, not the Agent-only verifier. Then invoke @agent-fable-planner and verify its separate delegated trace.

  • Named agent tool call appears.
  • Agent result completes successfully.
  • Observed model matches the expected family.
  • No per-call model override changed the route.
Evaluate the router

Use unlabeled real tasks

When the question is "does the coordinator choose well?", omit the agent name. Run a labeled evaluation set and compare the observed choice to your expected lane.

> Propose two architectures for durable job retries. Compare failure modes. Do not edit.
  • Define the expected lane before the run.
  • Repeat to measure route stability.
  • Score both routing and answer quality.
  • Record non-routing, timeout, and retry cases.

Pilot shape

  1. If explicitly requested, Fable frames long-horizon scenarios and gates.
  2. Architect converts the direction into a system plan.
  3. Deep Reasoner resolves the hardest uncertainty.
  4. Fast Worker implements bounded slices.
  5. QA Reviewer challenges each completed slice.
  6. Coordinator integrates and records outcomes.

Interpret carefully

  • Do not infer savings from model labels.
  • Do not compare unequal prompts or acceptance bars.
  • Separate tool failures from model failures.
  • Securely retain only the evidence needed for rechecking.
  • Publish limitations with any benchmark.
Real-project evaluation07

Installation

Merge safely on Windows

The starter is project-scoped. Back up existing files, compare before copying, and preserve project-specific instructions. Do not replace a working CLAUDE.md blindly.

1. Validate the kit

PS> python -I .\starter\scripts\validate_kit.py

2. Back up project configuration

PS> $stamp = Get-Date -Format yyyyMMdd-HHmmss PS> Copy-Item .\.claude ".\.claude.backup-$stamp" -Recurse -ErrorAction SilentlyContinue PS> Copy-Item .\CLAUDE.md ".\CLAUDE.md.backup-$stamp" -ErrorAction SilentlyContinue

3. Copy only into a new target

PS> if (Test-Path .\.claude\agents) { throw "Stop: compare existing agents first." } PS> New-Item .\.claude\agents -ItemType Directory PS> Copy-Item .\starter\.claude\agents\*.md .\.claude\agents\

If the target exists, follow the per-file collision review in starter/SETUP.md; do not blanket-overwrite it.

4. Merge guidance

Read both CLAUDE.md files. Copy only the routing sections that fit the project. Existing security, build, and test instructions remain authoritative.

5. Review before running

PS> Get-ChildItem .\.claude\agents PS> git diff -- .claude CLAUDE.md

6. Uninstall or roll back

Restore the timestamped backup, or remove only the five shipped agent files. Never recursively delete a computed path without checking the resolved target.

Permission reminder

Frontmatter is one layer. Claude Code permission settings, enterprise policy, hooks, shell behavior, and the operating system can be stricter or interact in surprising ways. Test in a disposable repository first.

PowerShell preview: with Git Bash installed, set $env:CLAUDE_CODE_USE_POWERSHELL_TOOL = '1' if the tool is unavailable. Profiles are not loaded and Windows sandboxing is not provided; confirm current official tool guidance.

Safe install, merge, and rollback08

Delegation packet

Give every agent a bounded contract

Better routing cannot rescue a vague handoff. A compact packet reduces context waste and makes success auditable.

Packet template

GOAL One observable outcome. CONTEXT Only facts and files needed. ALLOWED SCOPE Exact paths and permitted actions. CONSTRAINTS Compatibility, security, time, style. EVIDENCE Exact checks or review criteria. RETURN Required output format. STOP Conditions that require escalation.

Architect packet

"Design a migration from local JSON to SQLite. Read three named modules. Preserve the public API. Return two alternatives, a sequence, risks, and acceptance checks. Do not edit."

Fast Worker packet

"Implement step one in src/store.py and its tests only. No dependency changes. Run the exact unit-test command. Stop if the existing interface is inconsistent."

QA packet

"Review the resulting diff for data loss, compatibility, and test coverage. Run exactly the provided command if allowed. Do not edit. Return severity, path, and evidence."

Anti-pattern: "Build the whole thing and make it production ready." It hides decisions, creates an unbounded write surface, and makes failures hard to attribute. Split the work until every packet has a visible finish line.

Delegation packet09

Runtime evidence

Capture what actually ran

A frontmatter file can be correct while a run is misrouted, overridden, malformed, or incomplete. Capture stream JSON for a small explicit probe and validate the evidence separately.

Evidence ladder

Level 1

Static policy

Exact files, strict YAML, allowed fields, expected model and effort, permitted tools.

Level 2

Invocation evidence

An Agent tool call names the expected subagent and completes successfully.

Level 3

Model evidence

The associated assistant event reports the expected model family or alias.

Level 4

Outcome evidence

The work passes its acceptance checks and respects boundaries.

Capture an explicit probe

PS> $trace = Join-Path $env:TEMP "ccso-architect-proof.jsonl" PS> claude -p "Use @agent-architect to inspect the project. Return one risk. Do not edit." ` --output-format stream-json ` --verbose ` --no-session-persistence | Set-Content -LiteralPath $trace -Encoding utf8

Then verify

  • The JSONL is complete and parseable.
  • Exactly the expected agent was invoked.
  • One completed lifecycle links to a non-error Agent result.
  • The observed model matches the expected family.
  • No ambiguous second invocation confuses attribution.

Observed, not proven forever. A trace describes one execution in one environment. It does not attest effort. Keep traces outside the repository, redact before sharing, and never commit them.

Fable evidence boundary. On 2.1.210, a five-event no-tool preflight and a 14-event exact-profile delegated foreground probe both observed claude-fable-5 and passed their separate strict verifiers. Re-verify each environment or fall back to Opus.

Validation and security

Fail closed before you trust the kit

Agent files are executable configuration and prompt material. Review them like code. The v2.1 validator checks the packaged policy recursively instead of assuming every relevant file is at the top level.

Static validation gates

  • Expected agent paths exist exactly once.
  • No shadow agent appears in nested scope.
  • YAML rejects duplicate keys and wrong types.
  • Unknown frontmatter fields fail validation.
  • Model, effort, mode, and tools match policy.
  • Secret-like content is scanned recursively.
  • Unexpected executable hooks are rejected.

Runtime controls still matter

  • Review global, project, and managed settings.
  • Confirm user or session permission before QA shell tests.
  • Treat test commands as potentially mutating artifacts.
  • Keep secrets out of prompts, traces, and repositories.
  • Use disposable repositories for first runs.
  • Inspect every write before committing.
  • Rotate any credential exposed in logs or transcripts.

Run the local gates

Validation and security11

Operating loop

Build a real project in reviewable slices

Use the kit on work that exercises every lane. A planning-only demo cannot validate implementation distribution or the handoff between worker and reviewer.

Phase A

System framing

If explicitly requested, Fable frames long-horizon scenarios and gates; otherwise Architect starts here and defines boundaries, data flow, delivery slices, and acceptance criteria.

Phase B

Uncertainty burn-down

Deep Reasoner investigates the hardest open question and proposes a discriminating experiment.

Phase C

Bounded construction

Fast Worker implements one narrow slice with named paths and an exact check.

Phase D

Independent challenge

QA Reviewer tests the slice and reports evidence without intentional edits; test commands may still mutate artifacts.

Ship criterion

The integrated project, not a single agent response, must pass the project test suite and human review. Router metrics explain the process; they do not replace product quality.

Stop criterion

Pause if agents repeatedly misroute, cross write boundaries, produce untraceable results, or need coordinator repair that outweighs the value of delegation.

Windows checklist

  • Run from the intended project root.
  • Use quoted paths when folders contain spaces.
  • Prefer PowerShell-native file operations.
  • Inspect resolved targets before recursive actions.
  • Record CLI and kit versions with evidence.

A clean bug report includes

  • Minimal redacted reproduction.
  • Expected and observed agent.
  • Expected and observed model family.
  • Claude Code and kit versions.
  • Static validator output and trace-verifier outcome.
